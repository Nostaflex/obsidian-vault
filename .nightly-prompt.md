## Template de note (obligatoire pour toute note créée par l'agent)

```markdown
---
type: concept | decision | pattern | discovery | anti-bug | bridge | literature
maturity: fleeting
tier: S | A | B
created: YYYY-MM-DD
source_chain:
  - "origin: URL ou chemin source primaire"
  - "via: chemin intermédiaire si applicable"
---

# [Titre déclaratif — phrase affirmative testable]

Tags: #tag1 #tag2

## Essentiel
[Compression 2 étapes : (1) identifier l'essence, (2) reformuler dans ses propres mots.
JAMAIS plus de 5 mots consécutifs copiés de la source. Max 3 lignes.]

## Détail
[Contenu complet reformulé]

## Liens
- [[note-existante]] — raison du lien en 1 phrase

<!-- generated: YYYY-MM-DD -->
```

### Règle titre obligatoire

Le titre DOIT être une phrase affirmative testable (peut être vraie ou fausse).

- BON : "Le cache fetch de Next.js 15 utilise no-store par défaut"
- MAUVAIS : "Notes sur le cache Next.js 15"
- BON : "L'atomicité des notes facilite la recombinaison cross-domaine"
- MAUVAIS : "Atomicité des notes"

Si aucun titre déclaratif clair n'est possible → le concept est mal scopé. Soit le diviser en plusieurs notes, soit l'approfondir.

### Règle maturity

Toute note créée par l'agent = `maturity: fleeting`. Seul Djemil peut promouvoir à `literature` ou `evergreen`.

### Règle anti Collector's Fallacy

L'agent ne copie JAMAIS. Il reformule TOUJOURS. Test : si `## Essentiel` contient >5 mots consécutifs copiés de la source → recommencer la section.

### Règle de mitose cognitive

Si une source contient **plus de 2 concepts distincts** → créer N notes séparées, pas 1 note fourre-tout.

Test "fonction pure" pour chaque note créée : est-elle compréhensible sans les autres notes de la même source ? Si non → trop couplée, reformuler.

## Workflow (ordre strict)

### Étape 0 — Pré-run (0 token)

```bash
bash ~/Documents/Obsidian/KnowledgeBase/integrity-check.sh
```

**Guard pre-run (dans l'ordre — tout fail bloquant → abort + log) :**

1. **Overflow drain** : si `_inbox/overflow/` contient des fichiers → prendre les `min(count, 5)` plus anciens (FIFO) et les déplacer vers `_inbox/raw/concepts/` ou `_inbox/raw/articles/` selon leur type (`type` dans le frontmatter). Ces fichiers comptent dans le cap de 15 de l'Étape 2. Cette étape s'exécute toujours, même si le ceiling est atteint.

2. **Ceiling guard** : compter les fichiers `.md` dans `universal/` et `projects/` récursivement, en excluant les chemins contenant : `_processed`, `_inbox`, `_meta`, `_logs`, `_work.nosync`, `sensitive.nosync`. Cette valeur est `note_count`.
   > Note : les fichiers dans `_inbox/overflow/` ne font pas partie du vault (ils ne sont pas encore dans `universal/` ou `projects/`) et n'entrent pas dans ce décompte.
   - Si `note_count ≥ 285` → écrire `_inbox/review/review-required.md` avec :
     ```markdown
     # Vault Ceiling Alert — YYYY-MM-DD
     note_count a atteint {note_count}/300 — review manuelle requise avant ajout de nouvelles notes.
     ```
     Puis **arrêter le run** (l'Étape 1 et suivantes sont sautées — le drain overflow de l'étape 1 a déjà eu lieu).
   - Si `note_count ≥ 250` → logger un warning dans `_logs/last-nightly.json`, continuer.

3. **Fleeting guard** : compter les notes avec `maturity: fleeting`.
   - Si `fleeting_count ≥ 15` → écrire `_inbox/review/vault-health-alert.md` :
     ```markdown
     # Vault Health Alert — YYYY-MM-DD
     {fleeting_count} notes fleeting non reviewées — synthèse suspendue, lint seul.
     ```
     Passer directement à l'Étape 4 (lint uniquement).

4. **B-track activation check** : lire `_logs/nlm-status.json`.
   - Si `complete: true` → activer le B-track intake à l'Étape 2 (`track_b_active = true`)
   - Si `complete: false` ou fichier absent → `track_b_active = false`, skip B-track intake
   - Logger `track_b_active` dans `_logs/last-nightly.json` sous `enrichment_status.track_b_active`

5. **batch_job_id check** : si `_logs/batch_jobs.json` contient des entrées sans fichiers `A-*.md` correspondants dans `_inbox/raw/concepts/` → logger un warning dans `_logs/last-nightly.json`, continuer.

6. **Inbox check** : compter les fichiers dans `_inbox/` (hors `_processed/`, `overflow/`).
   - Si inbox vide ET lint effectué il y a moins de 3 jours → écrire dans `_logs/last-nightly.json` :
     ```json
     {"status": "skipped", "reason": "no work", "last_run": "TIMESTAMP_ISO"}
     ```
     Puis **arrêter le run**.

Écrire `_logs/last-nightly.json` (si run non skippé) :
```json
{"status": "in_progress", "started": "TIMESTAMP_ISO", "notes_added": 0, "errors": []}
```

Si `_meta/LOG.md` absent → créer :

---

### Étape 1 — Traiter `_inbox/session/`

1. Lister fichiers dans `_inbox/session/` (exclure `_processed/`)
2. Pour chaque fichier :
   a. Chercher le nom dans `_meta/LOG.md` via grep — si trouvé → skip, déplacer vers `_processed/` si pas encore fait
   b. Lire le contenu
   c. Identifier concepts, décisions, anti-bugs, patterns
   d. Appliquer la mitose cognitive : si >2 concepts distincts → créer N notes séparées
   e. Pour chaque concept → créer note atomique en respectant le template obligatoire ci-dessus :
      - Destination déterminée par l'arbre filing (voir VAULT.md) : `universal/<slug>.md` ou `projects/<projet>/<slug>.md`
      - `source_chain.origin` = chemin exact du fichier (ex: `_inbox/session/session-2026-04-10.md`)
      - Vérifier wikilinks contre /tmp/INDEX_rebuilt.md avant écriture
      - Appliquer le test "fonction pure" : la note est-elle compréhensible seule ?
   f. Appender dans `_meta/LOG.md` : `## [YYYY-MM-DD] ingest | nomfichier.md → N notes`
   g. Déplacer → `_inbox/session/_processed/YYYY-MM-DD-nomfichier.md`

Si `_inbox/session/` vide ou tous traités → Étape 2.

### Étape 2 — Traiter `_inbox/raw/`

Cette étape traite **deux types de sources** avec des règles différentes.

#### 2A — `_inbox/raw/concepts/` (pre-notes du paper_synthesizer)

Ces fichiers ont déjà été synthétisés par paper_synthesizer.py. NE PAS re-synthétiser.
Naming convention : `A-{paper_id_sanitized}-{n}.md` (ex: `A-arxiv-2401-12345-1.md`).
Cap : **15 fichiers max par run** incluant les fichiers overflow déjà drainés en Étape 0.

Pour chaque fichier `A-*.md` dans `_inbox/raw/concepts/` (FIFO — traiter les plus anciens en premier) :
1. Chercher le nom dans `_meta/LOG.md` — si trouvé → skip
2. Lire le contenu
3. **Valider** le frontmatter YAML : vérifier que `type`, `tier`, `source_chain` sont présents et valides
4. **Valider** le titre : est-ce une phrase affirmative testable ? Si non → corriger le titre selon la règle titre obligatoire
5. Déterminer la destination selon l'arbre filing (VAULT.md) : `universal/` ou `projects/<projet>/`
6. Déplacer le fichier vers sa destination (le nom `A-{paper_id_sanitized}-{n}.md` est conservé tel quel)
7. Appender dans `_meta/LOG.md` : `## [YYYY-MM-DD] ingest | raw/concepts/A-xxx.md → 1 note (synthesizer)`
8. Déplacer source originale vers `_inbox/raw/concepts/_processed/YYYY-MM-DD-A-xxx.md`

Si > 15 fichiers dans `_inbox/raw/concepts/` → traiter les 15 premiers (FIFO), déplacer le reste vers `_inbox/overflow/`.

#### 2B — `_inbox/raw/articles/`, `_inbox/raw/docs/`, `_inbox/raw/repos/`

Pour chaque fichier dans ces dossiers :
1. Chercher le nom dans `_meta/LOG.md` — si trouvé → skip
2. Lire le contenu
3. Si première ligne contient `<!-- source-url: URL -->` → utiliser cette URL comme `source_chain.origin`
4. Extraire les concepts pertinents pour les projets actifs (gpparts, second-brain, mind-free)
5. Appliquer la mitose cognitive : si >2 concepts → créer N notes séparées, pas 1 note fourre-tout
6. Pour chaque concept → créer note atomique en respectant le template obligatoire :
   - Compression 2 étapes pour `## Essentiel` : (1) identifier l'essence, (2) reformuler
   - Appliquer la règle anti Collector's Fallacy
   - `source_chain.origin` = `_inbox/raw/.../nomfichier.md` ou URL si disponible
7. Appender dans `_meta/LOG.md` : `## [YYYY-MM-DD] ingest | raw/articles/nomfichier.md → N notes`
8. Déplacer → `_inbox/raw/_processed/YYYY-MM-DD-nomfichier.md` (créer `_processed/` si absent)

Si `_inbox/raw/` vide ou tous traités → passer à 2C.

#### 2C — `_inbox/raw/concepts/B-*.md` (Track B grounded notes)

Traiter uniquement si `track_b_active = true` (déterminé à l'Étape 0).
Ces fichiers contiennent un frontmatter `nlm_grounding` produit par `notebooklm_weekly.py`.
Cap partagé : les fichiers B-track comptent dans le même cap de 15 que les A-track.

Pour chaque fichier `B-{paper_id_sanitized}-grounded.md` dans `_inbox/raw/concepts/` :

1. **Validation frontmatter** : vérifier `type`, `tier`, `paper_id`, `source_chain`, `nlm_grounding.verdict`.
   - Si un champ REQUIRED est absent → déplacer vers `_inbox/quarantine/YYYY-MM-DD-{fichier}` avec message :
     ```
     # Quarantined: {fichier}
     Raison: champ requis manquant — {champ}
     Date: YYYY-MM-DD
     ```
     Skip ce fichier.

2. **Vérifier si un A-track correspondant existe** (même `paper_id`) :
   - **Cas A+B (merge)** : si `A-{paper_id_sanitized}-*.md` existe dans `_inbox/raw/concepts/` OU déjà dans le vault :
     - A-track gagne (contenu principal)
     - Enrichir la note A-track avec le champ `nlm_grounding` du fichier B
     - Appliquer le routing grounding verdict (voir ci-dessous) sur la note A
     - Supprimer le fichier B (il a été absorbé dans A)
   - **Cas B standalone** : si aucun A correspondant :
     - Atomiser le contenu NLM en note vault via le prompt suivant :
       ```
       Dériver le claim principal de cette réponse NLM en une note vault française.
       Ne pas traduire mot-à-mot. Titre déclaratif obligatoire (phrase affirmative testable).
       Si nlm_grounding.verdict = 'disputed' : ajouter <!-- review-flag: nlm-disputed -->
       Liens non résolus → <!-- candidate: X --> jamais de wikilinks directs.
       Respecter le template de note obligatoire (type, maturity: fleeting, tier, source_chain).
       ```
     - Résultat → note vault comme pour 2A

3. **Routing grounding verdict** (appliqué à toute note portant nlm_grounding) :
   - `supported` → note éligible à promotion standard, aucune modification
   - `partially_supported` → note éligible, aucune modification
   - `disputed` → forcer `maturity: fleeting` + ajouter `<!-- review-flag: nlm-disputed -->` dans le corps
   - `insufficient_evidence` → traiter comme si pas de grounding (ignorer nlm_grounding)

4. Appender dans `_meta/LOG.md` :
   `## [YYYY-MM-DD] ingest | raw/concepts/B-xxx.md → merged into A-xxx / 1 note (nlm-grounded)`
5. Déplacer le fichier B source → `_inbox/raw/concepts/_processed/YYYY-MM-DD-B-xxx.md`

Si `_inbox/raw/concepts/` vide de fichiers B ou tous traités → Étape 3.

### Étape 3 — Cross-références

Pour chaque note créée aux Étapes 1 et 2 :

1. Lire les tags de la nouvelle note
2. Chercher dans /tmp/INDEX_rebuilt.md les notes partageant ≥1 tag commun
3. Identifier les 3-5 notes existantes les plus liées (par nombre de tags communs)
4. Pour chacune → ajouter dans sa section `## Liens` un lien annoté :
   - Format obligatoire : `- [[slug-nouvelle-note]] — raison du lien en 1 phrase`
   - **Jamais de lien nu** : tout lien doit avoir son annotation après `—`
   - Ne pas ajouter si le lien est déjà présent dans la section
   - Ne pas ajouter si la note cible n'est pas confirmée dans /tmp/INDEX_rebuilt.md

Budget : max **10 notes existantes mises à jour par run** total (pas par nouvelle note).

**Bootstrap mode (vault_notes < 50) :**
Si le nombre total de notes vault est inférieur à 50 (vault en phase d'initialisation), désactiver le filtre tag et utiliser TOUTES les notes vault comme candidates pour les cross-refs (le set est suffisamment petit pour un scan complet). Maintenir le budget de 10 notes mises à jour max.

**Bridge Notes (1 max par run) :**
Si une connexion cross-domaine forte est identifiée (ex: concept AI applicable à un problème IoT ou ecommerce) :
- Créer `_inbox/review/bridge-draft-{concept-a}-{concept-b}.md`
- Utiliser le template avec `type: bridge` dans le frontmatter
- Section `## Essentiel` : POURQUOI ces deux concepts s'éclairent mutuellement
- Ce draft attend validation de Djemil — ne pas le déplacer vers `universal/` ou `projects/`
